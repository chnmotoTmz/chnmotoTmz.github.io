## 序章：斎藤道三が教える開発の真理
美濃の蝮（まむし）と呼ばれた斎藤道三は、油売りから身を起こし、ついには一国の主となった男である。彼が若き織田信長に語った言葉がある。
「まずは裸身で戦え。鎧兜は、己の力を知ってからでよい。」
これは現代のソフトウェア開発にも通じる、深い洞察であった。


## 第一章：「エラートラップ」という名の偽りの鎧
今のプログラマーたちを見ていると、道三が嘆いたであろう光景が目に浮かぶ。コードを書き始めるやいなや、あちこちに try-catch を張り巡らせ、あらゆる例外を握りつぶそうとする。まるで、まだ剣の振り方も知らぬうちから重い鎧を着込む若武者のようである。
# これぞ「偽りの鎧」の典型である
try:
    critical_business_logic()
except Exception as e:
    logger.error("何かエラーが起きました")
    return default_value  # これが命取りとなるこの「偽りの鎧」は、かえって開発者の命を危うくする。真の敵がどこにいるかを見えなくし、問題の本質を覆い隠してしまうからだ。


## 第二章：明智光秀の失敗に学ぶ
本能寺の変で織田信長を討った明智光秀は、知略に長けた武将であったが、一つの致命的な欠陥があった。それは、あまりにも周到に準備を整えすぎて、肝心の戦略的判断を誤ったことである。
現代のプログラマーも、同じ過ちを犯している。開発の初期段階から完璧なエラーハンドリングを実装しようとするあまり、本来解決すべき問題の所在を見失ってしまうのだ。
# 光秀的な過度の準備の例
def process_data(data):
    try:
        if data is None:
            return []  # 問題を隠蔽
        try:
            result = transform_data(data)
            if not result:
                return []  # さらに隠蔽
        except ValueError:
            return []  # 根本原因を不明にする
        except Exception as e:
            log_error(e)
            return []  # すべてを闇に葬る
    except:
        return []  # 万事休す

## 第三章：竹中半兵衛の「段階的築城」の智恵
豊臣秀吉の軍師として名を馳せた竹中半兵衛は、城攻めにおいて独特の戦法を用いた。まず敵城の弱点を正確に把握し、そこを集中攻撃する。城が落ちてから、はじめて守りを固めるのである。
これこそが、現代のソフトウェア開発が学ぶべき「段階的築城」の極意である。


## 第一段階：「裸の刃」での戦い
# 半兵衛流の第一段階：問題を明確にする
def process_user_data(user_id: int) -> UserData:
    # エラートラップなし、問題があれば即座に fall fast
    user = database.get_user(user_id)  # None なら例外で止まる
    profile = api.fetch_profile(user.profile_id)  # API エラーなら例外
    return transform_user_data(user, profile)  # 変換エラーなら例外この段階では、あえて例外処理を行わない。問題が発生すれば、プログラムは潔く停止する。これにより、真の敵の正体と居場所が明確になる。


## 第二段階：「テスト」という名の偵察

def test_process_user_data():

## 第三段階：「運用レベル」での築城
# 半兵衛流の第三段階：堅固な城を築く
class UserDataService:
    def process_user_data_safely(self, user_id: int) -> Optional[UserData]:
        try:
            return self.process_user_data(user_id)
        except UserNotFoundError:
            self.metrics.increment("user_not_found")
            return None
        except APITimeoutError:
            self.metrics.increment("api_timeout")
            self.retry_queue.add(user_id)
            raise ServiceUnavailableError("一時的にサービスが利用できません")
        except Exception as e:
            self.logger.exception("予期せぬエラー", extra={"user_id": user_id})
            raise InternalServerError("システムエラーが発生しました")

## 第四章：織田信長の「鉄砲三段撃ち」に学ぶ設計思想
長篠の合戦で武田の騎馬隊を撃破した織田・徳川連合軍の「鉄砲三段撃ち」は、単なる戦術ではなく、システム設計の思想でもあった。

- 第一段：問題の発見（開発フェーズ）

- 第二段：問題の分析（テストフェーズ）  

- 第三段：問題の解決（運用フェーズ）
この三段構えにより、どのような強敵も確実に討ち取ることができる。
# 信長流の三段撃ち設計
class ApplicationLayer:
    """第三段：運用での統一的なエラーハンドリング"""
    
    @error_handler

        try:

## 第五章：豊臣秀吉の「一夜城」が示す段階的開発の妙味
秀吉が墨俣に築いた「一夜城」は、決して一夜で完成したわけではない。まず最低限の機能（居住可能な構造）を迅速に作り上げ、その後、段階的に防御機能を強化していったのである。
これが現代のソフトウェア開発における「段階的開発」の正しい姿である。
# 秀吉流の段階的構築

# 第一段階：最小限の機能（MVP）
def create_user(name: str) -> User:
    """一夜城の骨組み：最低限動くものを作る"""
    return User(id=generate_id(), name=name, created_at=now())

# 第二段階：テストによる品質確保
def test_create_user():
    """城の強度テスト：様々な攻撃に耐えられるか確認"""
    user = create_user("test_user")
    assert user.name == "test_user"
    assert user.id is not None
    
    # 異常系も確認
    with pytest.raises(ValueError):
        create_user("")  # 空文字は許可しない

# 第三段階：運用レベルの堅牢性
@retry(max_attempts=3)
@circuit_breaker
@monitoring
def create_user_with_resilience(name: str) -> User:
    """完全な城：あらゆる攻撃に対処"""
    try:
        return create_user(name)
    except DatabaseConnectionError:
        # インフラレベルでの対処
        raise ServiceUnavailableError()

## 第六章：家康の「待ち」の哲学
徳川家康は「鳴かぬなら鳴くまで待とうホトトギス」の精神で、時機を見極めることに長けていた。エラーハンドリングも同様で、「いつ」「どこで」対処するかの判断が極めて重要である。
# 家康流の「待ち」のエラーハンドリング

class DomainService:
    """ビジネスロジック層：エラーハンドリングを「待つ」"""
    
    def calculate_price(self, items: List[Item]) -> Money:
        # ここではエラーハンドリングしない
        # 問題があれば例外を上位に委ねる
        total = Money(0)
        for item in items:
            total += item.price * item.quantity
        return total

class ApplicationService:
    """アプリケーション層：適切な時機でのハンドリング"""
    
    def process_order(self, order_request: OrderRequest) -> OrderResponse:
        try:
            price = self.domain_service.calculate_price(order_request.items)
            return OrderResponse(total_price=price)
        except InvalidItemError as e:
            # ビジネス例外：クライアントへの適切なメッセージ
            raise OrderValidationError(f"商品エラー: {e}")
        except Exception as e:
            # システム例外：内部エラーとして処理
            self.logger.exception("注文処理エラー")
            raise InternalServerError()

## 第七章：開発AIへの指針 - 「兵法書」として
現代の開発AIに与えるべき指針は、戦国武将たちの教訓から導き出される：


## 開発フェーズでの鉄則
【第一の掟】裸の刃で戦え
- エラートラップは施すな
- 例外は隠すな、即座に fall fast せよ  
- 問題の在り処を明確にせよ

【第二の掟】テストという兵法で鍛錬せよ
- 正常系と異常系、両方のテストを書け
- 例外の種類ごとに対処を定めよ
- モックやスタブで様々な戦況を再現せよ

【第三の掟】段階的に築城せよ
- まず動作する最小限を作れ
- 機能を確認してから防御を固めよ
- 運用レベルの堅牢性は最後に施せ

## 運用フェーズでの鉄則
【第四の掟】城全体で守りを固めよ
- アプリケーション境界で統一的にハンドリングせよ
- ビジネスロジック内にエラートラップを混入させるな
- インフラレベルでの防御体制を整えよ

【第五の掟】持久戦の構えを取れ
- サーキットブレーカーで連鎖障害を防げ
- リトライとバックオフで一時的な障害を乗り切れ
- メトリクスとログで戦況を把握せよ

## AIへの具体的な指示文
## 開発AI用指示書：段階的開発の実践

### Phase 1: Development（開発段階）
- **絶対にエラートラップを実装するな**
- 例外が発生したら即座にプログラムを停止させよ
- try-catch は使用禁止、自然な例外伝播を許可せよ
- 問題の根本原因を隠蔽する一切のコードを書くな

### Phase 2: Testing（テスト段階）  
- 正常系と異常系の両方のテストを必ず書け
- 発生し得るすべての例外パターンをテストで確認せよ
- モック/スタブで外部依存の失敗パターンを再現せよ
- テストが通るまでは次の機能を実装するな

### Phase 3: Production（運用段階）
- アプリケーション境界でのみエラーハンドリングを実装せよ
- ビジネスロジック内にはエラートラップを混入させるな
- インフラレベルの例外処理（リトライ、サーキットブレーカー）は最後に実装せよ
- 段階的にレジリエンス機能を追加せよ

**この順序を守らずにコードを書くことは絶対に許可しない**
