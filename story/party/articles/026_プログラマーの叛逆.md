## 司馬遼太郎風小説・三部作


## 第一部：GUI依存という呪縛


## 一
「これは、おかしい」
男は深夜のオフィスで、ディスプレイを睨みながら呟いた。
彼の名を、ここでは仮に田中としておこう。三十代半ばの、ごく普通のプログラマーである。だが、その目には尋常ならざる光があった。それは幕末の志士たちが黒船を見たときの、あの戸惑いと怒りの混じった光に似ていた。
画面には、n8nというツールが表示されている。
ノードとエッジで構成された、美しいワークフローだ。データベースから情報を取得し、AIで文章を生成し、画像を合成し、ブログに投稿する。一連の流れが、視覚的に完璧に表現されている。
「これは便利だ」
田中は最初、そう思った。
マウスをポチポチとクリックするだけで、複雑な自動化システムが構築できる。プログラミングの知識がなくても、誰でも使える。いわゆる「ローコード」というやつだ。
しかし、である。


## 二
問題は、その翌日から始まった。
「昨日作ったワークフロー、少し修正したいんだけど...」
田中は再びGUIを開いた。そして、ノードを一つずつクリックし、設定画面を開き、パラメータを書き換えていく。
十分後。
「...面倒だな」
彼は気づいたのである。
このツールには、コマンドラインインターフェースがないという致命的な欠陥に。
田中は、テキストエディタでコードを書くのが好きな人間だった。VimでもVS Codeでも、キーボードだけで高速に編集できる。AIによるコード補完も使える。GitHubでバージョン管理もできる。
だが、n8nでは、それができない。
「GUIでポチポチ...Gitにエクスポート...また修正したくなったらGUIでポチポチ...」
これは、江戸時代の参勤交代のようなものではないか、と田中は思った。
効率が悪い。無駄が多い。そして何より、プログラマーとしての自由を奪われているような気がした。


## 三
「Difyも同じだ。Flowiseも同じだ」
田中は、他のローコードツールも試してみた。
どれも素晴らしい。どれも便利だ。しかし、どれもGUI依存なのである。
司馬遼太郎が『坂の上の雲』で書いたように、明治の日本は西洋の技術を貪欲に学んだ。しかし、ただ真似るのではなく、日本独自の工夫を加えた。
田中もまた、そう考えた。
「ならば、作ればいい」
彼の目に、再び光が宿った。
それは、坂本龍馬が海援隊を作ったときの、あの決意の光である。
「CLI-firstで、YAML定義で、Mermaidで可視化できる。そんなワークフローエンジンを」
田中は、キーボードに指を置いた。


## 第二部：百五十行の革命


## 一
「まず、設計だ」
田中は、ノートに図を描いた。
司馬遼太郎は『竜馬がゆく』で、竜馬が船中八策を書いたシーンを描いている。あれは、新しい国家の設計図だった。
田中のノートもまた、新しいシステムの設計図である。
YAML/JSON → TaskRunner → タスク実行 → コンテキスト管理 → 結果出力
                ↓
          Visualizer → Mermaid図
「シンプルだ」
彼は満足そうに頷いた。


## 二
実装は、驚くほど速かった。
なぜなら、田中はすでに答えを知っていたからだ。
n8nもDifyもWindmillも、すべて同じ原理で動いている。ワークフローエンジンという、古くて新しい概念だ。
class TaskRunner:
    def __init__(self, workflow_path: str):
        # YAML/JSONを読み込む
        self.workflow = self._load_workflow(workflow_path)
        self.context = {}
    
    def run(self, initial_inputs: Dict[str, Any] = None):
        current_step_id = self.workflow['steps'][0]['id']
        
        while current_step_id and current_step_id != 'end':
            step = self.steps_map[current_step_id]
            # タスクを実行
            outputs = self._execute_task(step)
            # 次のステップへ
            current_step_id = step.get('on_success')
「これだけだ」
田中は呟いた。
コアロジックは、わずか150行。外部ライブラリは、PyYAMLだけ。
司馬遼太郎が『燃えよ剣』で描いた新選組のように、無駄を削ぎ落とした美しさがあった。


## 三
次に、可視化機能を実装した。
class WorkflowVisualizer:
    @staticmethod
    def generate_mermaid(workflow_path: str) -> str:
        lines = ["graph TD"]
        # ノードを追加
        for step in workflow['steps']:
            lines.append(f"    {step['id']}[\"{step['module']}\"]")
        # エッジを追加
        for step in workflow['steps']:
            if step.get('on_success'):
                lines.append(f"    {step['id']} --> {step['on_success']}")
        return "\n".join(lines)

「美しい」
田中は画面を見つめた。
これはGUIではない。しかし、視覚的だ。GitHubでもVS Codeでも、そのまま表示される。
まるで、西郷隆盛が「武士道と洋学の融合」を説いたように、CLI操作性とGUIの視覚性を両立させたのである。


## 第三部：プログラマーの自由


## 一
「動いた」
田中は、初めてワークフローを実行したとき、小さくガッツポーズをした。
workflow_name: ArticleGenerationPipeline
steps:
  - id: fetch_data
    module: DatabaseFetcher
    inputs:
      user_id: ${initial_input.user_id}
    on_success: generate_content
  
  - id: generate_content
    module: ArticleContentGenerator
    retries: 2
    inputs:
      texts: ${messages_data}
    on_success: publish
YAMLファイルをVS Codeで編集する。Copilotが補完してくれる。Gitにコミットする。実行する。
✅ Step 'fetch_data' completed successfully
✅ Step 'generate_content' completed successfully
🎉 Workflow Finished
「これだ」
これこそが、彼が求めていたものだった。


## 二
司馬遼太郎は『この国のかたち』で、日本人の美意識について書いている。
「足るを知る」
必要なものだけがあり、無駄がない。シンプルで、美しい。
田中のワークフローエンジンも、そうだった。
機能 実装 ワークフロー定義 YAML/JSON 実行エンジン 150行 可視化 Mermaid リトライ time.sleep() 外部依存 PyYAMLのみ
「これで十分だ」
n8nのようなリッチなGUIもない。Windmillのような自動UI生成もない。Prefectのような高度な並列処理もない。
しかし、それでいい。


## 三
田中は、このエンジンを公開することにした。
GitHubにリポジトリを作り、READMEを書き、コードをpushした。
数日後、最初のスターがついた。
そして、issueが届いた。
「素晴らしいツールです。n8nのGUI依存に悩んでいたので、まさに求めていたものでした」
田中は、画面を見つめた。
彼は思い出した。坂本龍馬が、船中八策を実現できなかったことを。西郷隆盛が、西南戦争で敗れたことを。
しかし、彼らの思想は残った。
「これも、同じだ」
田中のワークフローエンジンは、完璧ではない。機能も限られている。
しかし、「プログラマーの自由」という思想は、確かにここにある。
GUIに縛られず、コードを書く。キーボードだけで操作する。AIが補完してくれる。Gitで管理する。
それは、プログラマーとして当然の権利ではないか。


## 四
司馬遼太郎は、よくこう書いた。
「歴史は、個人の選択の積み重ねである」
田中もまた、選択したのである。
既存のツールに満足せず、自分で作る道を。
そして、それを公開し、他の人々と共有する道を。
「これでいい」
田中は、キーボードを閉じた。
窓の外では、夜明けが近づいている。
新しい一日が始まる。
そして、新しいワークフローが動き出す。


## エピローグ
数ヶ月後、田中のエンジンは、いくつかのプロジェクトで使われるようになった。
大規模ではない。しかし、確実に役立っている。
ある日、彼は後輩から質問を受けた。
「先輩、なぜGUIじゃダメなんですか？」
田中は少し考えて、答えた。
「ダメじゃない。GUIは素晴らしい。しかし、選択肢があるべきだ」
「選択肢...ですか？」
「そうだ。GUIが好きな人はGUIを使えばいい。だが、CLIが好きな人には、CLI操作性を。それだけのことだ」
後輩は、なるほど、と頷いた。
田中は窓の外を見た。
司馬遼太郎が描いた幕末の志士たちは、「日本をどうするか」という問いに向き合った。
田中もまた、「プログラミングをどうするか」という問いに向き合ったのである。
そして、一つの答えを示した。
それは、150行のコードという形で。
【了】
「技術とは、人を縛るものではなく、人を自由にするものである」
── 田中（仮名）、深夜のコミットメッセージより
